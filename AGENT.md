# 相关概念
## 时延
在网络传输中，时延（delay 或 latency）是数据（一个报文或分组，甚至比特）从网络（或链路）的一端传送到另一端所需的时间。通常由发送时延、传播时延、处理时延、排队时延构成，其中：
  - （1）发送时延，也称为传输时延。指发送数据时，数据帧从结点进入到
传输媒体所需要的时间，即从发送数据帧的第一个比特算起，到该帧的最后一
个比特发送完毕所需的时间；与传输数据量的规模密切相关。
  - （2）传播时延，电磁波在信道中需要传播一定的距离而花费的时间；
  - （3）处理时延，主机或路由器在收到分组时，为处理分组（例如分析首
部、提取数据、差错检验或查找路由）所花费的时间；
  - （4）排队时延，是指分组在路由器输入输出队列中排队等待处理所经历的时延，其长短往往取决于网络中当时的通信量。
  
数据在网络中经历的总时延为发送时延、传播时延、处理时延和排队时延之和。

## 压缩控制
压缩控制是减小原始数据规模以减小传输量的一个有益思路，其通过特定算法或策略，在尽可能保留数据核心信息（或满足应用场景信息需求）
的前提下，减少数据存储体积、降低数据传输带宽消耗的技术。其本质是“去除数据冗余”，包括重复数据的冗余（如连续相同的数值）、统计规律的冗余（如数据出现概率的不均衡）、语义/视觉感知的冗余（如人眼无法分辨的细微图像
差异）等，最终实现数据轻量化处理，同时兼顾数据的可用性。因此在可能的
情况下，对原始监测数据进行压缩处理，无疑能够从根源上降低数据的规模，
减小数据传输和数据存储（缓存、暂存）的压力。

# 目标
实现一个应用程序用于支撑数据压缩算法的测试评估，实现服务端与客户端之间的数据传输通信，并输出每次通信的发送时延、传播时延、处理时延、排队时延等指标。程序内置多个数据压缩算法，可选择相应的压缩算法对数据进行压缩处理后发送，接收方接收到后做解压处理。

# 系统主流程
客户端与服务端连接，客户端选择要传输的文件和本次传输所使用的压缩算法，将文件压缩后，经网络发送给服务端。服务端接收到后，根据选定的压缩算法做解压，将收到数据的时间、解压耗费的时间返回给客户端，客户端将压缩耗时、发送耗时、解压耗时这些指标显示，方便比较不同压缩算法的性能。

# 实现要求
服务端和客户端都采用JAVA开发，内置的压缩算法至少支持GZIP,DEFLATE,ZIP,BZIP2,LZMA,LZ4,Zstd,SNAPPY,Brotli可供选择，并提供扩展接口方便后续扩充。
考虑通过JavaFX实现客户端UI界面，实时显示服务端和客户端之间的网络实时链接状态，用户选择要发送的文件以及本次发送所使用的压缩算法，点击发送后显示发送进度，并在发送完成收到响应后，计算出发送和接收的总字节数、详细耗时、时延分布、压缩比等指标，以衡量不同算法的性能差异。
